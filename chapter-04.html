5  Boundaries: From a Pile of Code to a Modular Map

In the previous chapters, we explored the fundamental differences between demo software and real software, and where the “maturity feeling” of professional software comes from. Behind these maturity signals, there is often a more fundamental structural element: boundaries.

边界是项目从练习走向专业的最早迹象之一。

在小型实验中，边界几乎不存在。开发者为了快速验证想法，将代码写在哪里方便就放在哪里。一个文件可能同时包含数据定义、业务逻辑、输入处理和输出格式化。函数之间自由调用，职责混乱不清。

最初，这种方式似乎完全可行。项目规模小，开发者理解整个系统，目标只是让想法运行起来。

然而，一旦项目开始增长——哪怕只是轻微增长——边界的缺失就会开始制造摩擦。变更变得难以理解，代码感觉纠缠不清，新功能需要修改多个不相关的区域。

这就是系统必须从一堆代码演变为模块化地图的关键时刻。

5.0.1 Why Boundaries Matter

边界是系统在增长过程中保持可理解性的关键。

一个边界定义了一个组件负责什么、不负责什么。它限制了系统其他部分与该组件交互的表面积。边界就像是一堵半透明的墙——它允许必要的通信，但阻止了无控制的渗透。

没有边界，一切都与其他一切相连。依赖关系呈指数级增长，系统逐渐演变成一个密集的交互网络。在这样的系统中，任何微小的变更都可能产生连锁反应，影响系统中看似无关的部分。开发者不敢轻易修改代码，因为他们无法预知副作用会传播到哪里。

有了边界，系统变得可导航。每个模块都可以被独立理解，变更保持局部化，而不是扩散到整个代码库。当一个模块的职责清晰定义时，开发者可以自信地修改它，因为他们知道影响范围被限制在边界之内。

换句话说，边界将代码从一堆指令集合转变为一个结构化环境。在这个环境中，每个部分都有明确的位置和职责，整体呈现出有序的组织而非混乱的纠缠。

这种转变不仅影响代码的可维护性，也影响开发者的认知负担。在有边界的系统中，开发者只需要同时理解少数几个相关模块，而不是整个系统。认知负荷的降低使得复杂系统变得可管理。

5.0.2 Module Boundaries

The most visible form of boundary is the module.

A module groups related functionality into a coherent unit. It provides a clear place where certain responsibilities live and prevents them from leaking into unrelated areas.

For example, a system that manages tasks might naturally divide into modules such as:

  • core logic that defines how tasks behave

  • storage mechanisms that persist data

  • interfaces that allow users to interact with the system

Each module contains code that serves its specific purpose. The storage module does not attempt to render user interfaces, and the interface module does not implement storage algorithms.

This separation may appear simple, but it fundamentally changes how the system is understood. The project begins to resemble a map rather than a maze.

5.0.3 Responsibility Boundaries

Closely related to modules are responsibility boundaries.

A responsibility boundary answers a simple question: what is this component supposed to do?

When responsibilities are clearly defined, each module has a focused role. A component that manages data persistence does not suddenly start performing complex business logic. A user interface component does not attempt to enforce domain rules.

Practice projects often blur these boundaries. A single function might read input, validate it, store the result, and print output—all in one place. The logic works, but the responsibilities are mixed together.

Professional systems separate these concerns deliberately. Each component focuses on a single responsibility, and interactions between components remain explicit.

This separation reduces complexity because developers can reason about each piece independently.

5.0.4 Dependency Direction

Another essential aspect of boundaries is the direction of dependencies.

In a well-structured system, dependencies flow in a consistent direction. Higher-level components depend on lower-level ones, or outer layers depend on inner layers, but the reverse relationship is avoided.

For example, interface components might depend on the core domain logic, while the core logic remains independent of the interface. This ensures that changes to presentation layers do not affect the fundamental behavior of the system.

When dependency direction becomes inconsistent, the system begins to lose its structure. Modules start referencing each other in cycles, and it becomes difficult to change one component without affecting many others.

Maintaining clear dependency direction preserves the integrity of the system’s architecture.

5.0.5 The Folder Structure as System Topology

Many developers treat directory structures as cosmetic choices. They organize files into folders primarily to reduce visual clutter.

In mature systems, however, the folder structure carries a deeper meaning.

The directory layout represents the topology of the system—a map of how components relate to each other. Each folder signals a boundary. Each layer reflects a specific responsibility.

For example, a project might organize itself around major roles within the system:

  • a core layer that contains domain logic

  • a storage layer that manages persistence

  • an interface layer that exposes commands or APIs

When this structure is consistent, the project becomes self-explanatory. A developer exploring the repository can quickly infer how the system is organized simply by observing the directory layout.

In this sense, folders are not decorative. They are architectural markers.

5.0.6 From Code to Architecture

从无结构代码到模块化架构的转变并不需要巨大的复杂性。事实上，最有效的边界往往简单而直观。

重要的不是模块的数量，而是它们之间关系的清晰度。

当一个项目定义了清晰的模块边界、职责边界和依赖方向时，重要的事情发生了：代码库变得可导航。

开发者可以从任何点进入系统，并快速定位自己。新功能可以在不影响项目无关部分的情况下添加。重构变得更安全，因为结构揭示了变更应该发生的位置。

这种转变标志着向专业软件迈出的最早步骤之一。

一个曾经像脚本集合的项目，现在开始像一个有地图的系统。

5.1 Practical Guide: How to Define Boundaries

将理论转化为实践需要具体的方法。以下是一个简化的边界划分SOP，可以帮助你将混乱的代码组织成清晰的模块结构。

5.1.1 边界划分SOP（标准操作流程）

第一步：识别职责

  1. 列出你的系统当前做的所有事情
  2. 将相似的功能分组
  3. 为每个组定义一个简洁的职责描述

第二步：绘制模块图

  1. 画出代表每个模块的框
  2. 用箭头表示模块间的依赖关系
  3. 确保依赖关系是有方向的（避免循环依赖）

第三步：定义接口

  1. 为每个模块定义公开的接口
  2. 明确每个接口的输入和输出
  3. 文档化接口契约

第四步：重构代码

  1. 将相关代码移动到对应的模块
  2. 删除跨模块的直接访问
  3. 确保所有通信都通过定义的接口

第五步：验证

  1. 检查是否还有跨模块的直接访问
  2. 验证依赖方向是否一致
  3. 确保每个模块都有单一、清晰的职责

5.1.2 常见边界反模式

在划分边界时，避免以下常见错误：

反模式1：万能模块 - 症状：一个模块负责太多不相关的功能 - 问题：难以理解、难以测试、难以维护 - 解决方案：按照单一职责原则拆分

反模式2：循环依赖 - 症状：模块A依赖模块B，模块B又依赖模块A - 问题：系统难以独立测试和部署 - 解决方案：提取共享依赖到新模块

反模式3：过度细分 - 症状：将系统拆分成过多小模块，每个模块功能过于简单 - 问题：增加复杂度，难以理解整体结构 - 解决方案：合并功能密切相关的模块

反模式4：隐式边界 - 症状：边界只存在于开发者的头脑中，代码中没有明确体现 - 问题：新开发者无法理解结构 - 解决方案：通过包结构、接口和文档明确边界

5.3 Chapter Summary

边界是专业软件的基础。它们将混乱的代码组织成有序的结构，使系统在增长时保持可理解性和可维护性。

通过定义模块边界、职责边界和依赖方向，我们将代码从一堆指令转变为一个可导航的系统。

下一章，我们将探讨另一个关键的结构性要素：命名。好的命名不仅仅是标签，它是系统的领域语言，直接影响代码的可读性和专业感知。

实践建议：

  1. 从小处开始：不要试图一次性重构整个系统。选择一个清晰的边界，先在那里实施，观察效果。
  2. 文档先行：在写代码之前，先画出模块图和定义接口。这会帮助你更清晰地思考结构。
  3. 保持简单：最有效的边界往往是简单的。不要为了追求理论完美而过度设计。
  4. 持续验证：定期检查是否出现了新的边界违规。边界需要持续维护。

立即行动：

选择你当前项目中的一个文件，尝试识别其中包含的不同职责。你能将它拆分成多个更小、更专注的模块吗？
