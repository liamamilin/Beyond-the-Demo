13  Configuration and Environments: From Hard-Coded Values to Deployable Systems

Many practice projects begin with a simple assumption: the code will run in a single environment controlled entirely by the developer.

Because of this assumption, important values are often written directly into the code. Database addresses, API keys, file paths, and service endpoints may all appear as fixed constants inside the program.

At first, this approach seems convenient. Everything needed to run the system exists in one place, and the developer does not need to worry about external configuration.

However, as soon as the system moves beyond a single machine or a single developer, this approach quickly becomes a limitation.

Professional systems solve this problem by separating configuration from code. Instead of embedding environment-specific values inside the program, they allow those values to be defined externally.

This separation is one of the essential steps that transforms a practice project into software that can be deployed reliably.

13.0.1 The Problem with Hard-Coded Values

Hard-coded values are easy to introduce during early development. A developer may temporarily specify a database connection string or an API endpoint directly inside the code.

While this may seem harmless, these embedded values create hidden dependencies.

If the system needs to run in a different environment—perhaps on another machine, a testing server, or a production platform—the code itself must be modified. Each modification introduces risk and increases the likelihood of configuration errors.

Hard-coded values also make it difficult to share the project safely. Sensitive information such as credentials may accidentally be committed to version control, exposing security risks.

Professional systems avoid these problems by treating configuration as an external concern rather than an internal detail.

13.0.2 Configuration as a Separate Layer

A mature system distinguishes between logic and configuration.

The logic describes how the system behaves. Configuration describes how the system connects to its environment.

Common configuration elements include:

  • database connection details

  • service endpoints

  • authentication credentials

  • feature flags

  • runtime parameters

Instead of embedding these values directly in the code, professional systems load them from configuration sources such as environment variables or configuration files.

This separation allows the same codebase to operate under different conditions without modification.

The software becomes portable. It can move from one environment to another simply by adjusting its configuration.

13.0.3 Environment Variables

One of the most common mechanisms for managing configuration is the use of environment variables.

Environment variables allow external values to be injected into the application at runtime. These values can vary depending on the machine or environment in which the software is running.

For example, a system might read the address of its database or the location of an external service from environment variables.

Because these values are not stored inside the codebase, they can be managed securely and adjusted independently of the application logic.

Environment variables also integrate naturally with deployment platforms and automation tools, making them a widely adopted standard in professional systems.

13.0.4 Configuration Profiles

As systems grow more complex, they often operate in multiple environments. Each environment may require slightly different configuration settings.

For example, a development environment might use a local database and detailed debugging output. A testing environment might simulate production conditions while isolating test data. A production environment might prioritize performance and security.

To manage these differences, many systems introduce configuration profiles.

A profile groups together configuration values appropriate for a particular environment. When the application starts, it selects the profile that corresponds to its current environment.

This approach allows developers to maintain a single codebase while adapting behavior to different operational contexts.

13.0.5 The Minimal Environment Model

Although large systems may define many environments, most projects require only a small number of layers to function effectively.

A minimal and widely used structure includes three environments:

Development
The environment used by developers during active coding. Debugging tools and detailed logging are often enabled here.

Staging
A testing environment that closely resembles production. New features are validated here before they are released publicly.

Production
The environment used by real users. Stability, performance, and security are prioritized.

This simple separation provides a clear progression from experimentation to real-world use. Changes can be tested safely before reaching the production environment.

Even small projects benefit from this structure once they begin interacting with real users or external systems.

13.0.6 From Local Scripts to Deployable Systems

The introduction of configuration and environment management represents a subtle but important shift in perspective.

Practice projects assume a single environment and a single user. Professional systems assume multiple environments and many users.

By separating configuration from code and organizing environments into clear layers, developers create software that can move reliably from development machines to staging servers and eventually to production systems.

This transformation may not change the visible functionality of the software, but it dramatically increases the system’s flexibility and reliability.

The project is no longer tied to one specific machine or context.

It becomes deployable.

13.1 Practical Guide: Configuration Management SOP

配置管理是将演示项目转化为可部署系统的关键步骤。以下是一个实用的配置管理框架，帮助你系统化地管理配置。

13.1.1 配置项分类

敏感配置（必须使用环境变量）： - 数据库密码 - API密钥和Token - 加密密钥 - 第三方服务凭证

环境特定配置（使用配置文件）： - 数据库连接地址 - 服务端点URL - 日志级别 - 功能开关

运行时参数（使用配置文件）： - 端口号 - 超时时间 - 批处理大小 - 并发限制

13.1.2 配置管理SOP

第一步：识别配置项

  1. 搜索代码中的硬编码值
  2. 列出所有可能变化的参数
  3. 分类为敏感配置、环境配置、运行时参数

第二步：选择配置方式

# 决策树：如何选择配置方式？
这个配置是否包含敏感信息？
├─ 是 → 使用环境变量（推荐）
│   └─ 示例：DATABASE_PASSWORD, API_KEY
└─ 否 → 这个配置是否在不同环境间变化？
    ├─ 是 → 使用配置文件（推荐）
    │   └─ 示例：config.dev.yaml, config.prod.yaml
    └─ 否 → 考虑使用默认值
        └─ 示例：端口号、超时时间

第三步：创建配置模板

.env.example 文件模板：

# 数据库配置
DATABASE_URL=postgresql://user:password@localhost:5432/mydb
DATABASE_PASSWORD=your_password_here

# API配置
API_KEY=your_api_key_here
API_ENDPOINT=https://api.example.com

# 功能开关
FEATURE_NEW_UI=false
FEATURE_ANALYTICS=true

config.yaml 配置文件模板：

server:
  host: localhost
  port: 8000
  debug: true

database:
  host: localhost
  port: 5432
  name: myapp_dev

logging:
  level: DEBUG
  format: json

features:
  new_ui: false
  analytics: true

第四步：实现配置加载

# Python示例：配置加载器
import os
from pathlib import Path
from typing import Any
import yaml

class Config:
    def __init__(self, env: str = "development"):
        self.env = env
        self.load_from_file()
        self.load_from_env()
        self.validate()

    def load_from_file(self):
        """从配置文件加载配置"""
        config_path = Path(f"config.{self.env}.yaml")
        if config_path.exists():
            with open(config_path) as f:
                self.config = yaml.safe_load(f)

    def load_from_env(self):
        """从环境变量加载配置（优先级更高）"""
        self.config["database"]["password"] = os.getenv(
            "DATABASE_PASSWORD",
            self.config.get("database", {}).get("password", "")
        )
        self.config["api"]["key"] = os.getenv(
            "API_KEY",
            self.config.get("api", {}).get("key", "")
        )

    def validate(self):
        """验证配置的完整性"""
        required_keys = [
            "database.url",
            "api.key",
        ]
        for key in required_keys:
            if not self.get(key):
                raise ValueError(f"Missing required configuration: {key}")

    def get(self, key: str, default: Any = None) -> Any:
        """获取配置值"""
        keys = key.split(".")
        value = self.config
        for k in keys:
            value = value.get(k, {})
        return value if value != {} else default

# 使用示例
config = Config(env=os.getenv("ENV", "development"))
db_url = config.get("database.url")

第五步：配置安全措施

  1. 将 .env 文件添加到 .gitignore
  2. 提供清晰的 .env.example 模板
  3. 实施配置验证机制
  4. 使用密钥管理服务（生产环境）

13.1.3 配置安全检查清单

安全措施： - [ ] 敏感信息使用环境变量，不写入配置文件 - [ ] .env 文件已添加到 .gitignore - [ ] 提供完整的 .env.example 模板 - [ ] 配置有类型验证和默认值 - [ ] 不同环境有独立配置文件 - [ ] 生产环境配置不在代码库中

可维护性： - [ ] 配置文档清晰说明每个配置项的用途 - [ ] 配置变更需要代码审查 - [ ] 配置有版本控制（模板文件） - [ ] 配置错误有清晰的错误消息 - [ ] 支持配置热更新（可选）

13.2 Chapter Summary

实践建议：

  1. 尽早实施：不要等到部署时才考虑配置。从一开始就设计好配置管理策略。
  2. 文档先行：在添加配置项之前，先记录其用途和默认值。
  3. 安全优先：始终假设配置可能泄露，不要在配置中存储敏感信息的明文。
  4. 验证配置：在应用启动时验证配置的完整性，避免运行时错误。
