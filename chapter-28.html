29  Appendix

Practical systems are not built only from ideas.
They are built from repeatable structures.

The following appendices provide ready-to-use templates and quick references that help transform a small project into something that immediately feels more professional.

These templates are intentionally minimal.
Their goal is not complexity, but clarity and consistency.

30 A. Professional README Template (Copy-Ready)

A professional README answers three questions quickly:

  1. What is this?

  2. Why does it exist?

  3. How do I use it?

Below is a minimal structure used by many mature projects.

# Project Name

Short one-sentence description.

Example:
A lightweight CLI tool for managing personal expense logs with natural language.

---

## Features

- Feature 1
- Feature 2
- Feature 3

---

## Installation

```bash
pip install project-name
# or
npm install project-name

30.1 Quick Start

project-name init
project-name add "coffee 5"
project-name report

30.2 Example

Input:

add lunch 12

Output:

Expense recorded.
Total today: $12

30.3 Configuration

Environment variables:

PROJECT_DB_PATH
PROJECT_ENV

30.4 Architecture (Optional)

High-level system components:

CLI → Service Layer → Storage

30.5 Roadmap

  • Web dashboard

  • Mobile input

  • Export reports

30.6 Contributing

Pull requests are welcome.

30.7 License

MIT


A structured README alone can dramatically improve how a project is perceived.

# B. Directory Structure Templates

A clear directory structure communicates system architecture immediately.

Below are three common patterns.

---

## Python Project

project/
│
├─ src/
│ └─ project/
│ ├─ init.py
│ ├─ cli.py
│ ├─ service/
│ ├─ models/
│ └─ storage/
│
├─ tests/
│
├─ docs/
│
├─ examples/
│
├─ pyproject.toml
├─ README.md
└─ LICENSE


Core ideas:

- `src/` prevents import confusion
- `models/` holds domain entities
- `service/` contains business logic

---

## Node.js Project

project/
│
├─ src/
│ ├─ api/
│ ├─ services/
│ ├─ models/
│ ├─ middleware/
│ └─ utils/
│
├─ tests/
│
├─ scripts/
│
├─ package.json
├─ README.md
└─ .env.example


Principle:

Separate **API**, **logic**, and **data model**.

---

## Web Application

project/
│
├─ frontend/
│ ├─ components/
│ ├─ pages/
│ ├─ hooks/
│ └─ styles/
│
├─ backend/
│ ├─ api/
│ ├─ services/
│ └─ models/
│
├─ infra/
│
├─ docs/
│
└─ README.md


Principle:

Separate **UI**, **logic**, and **infrastructure**.

---

# C. UI Rules Cheat Sheet

Professional interfaces follow a few consistent visual rules.

### Typography

Common hierarchy:

Title 32px
Heading 24px
Subheading 18px
Body 16px
Caption 14px


Rule:

Never invent new sizes randomly.

---

### Spacing Scale

Use consistent increments.

Example:

4
8
16
24
32
48
64


Rule:

All margins and paddings should come from the scale.

---

### Color System

Minimal palette example:

Primary
Gray 900 (text)
Gray 700
Gray 500
Gray 300
Gray 100 (background)

Accent color
Error color
Success color


Rule:

Most UI should use grayscale.  
Accent colors are used sparingly.

---

### Alignment

Always align elements to a grid.

Avoid random offsets.

---

### White Space

If a page feels messy:

Add spacing before adding design.

---

# D. Release Checklist

Before publishing a release, professional teams verify a small set of conditions.

Minimal release checklist:

### Code

- Tests pass
- Linting passes
- No debug logs

---

### Documentation

- README updated
- Changelog updated
- Version number updated

---

### Packaging

- Install command works
- Dependencies correct

---

### Security

- Secrets removed
- `.env` not committed

---

### UX

- Empty state works
- Error state works
- Loading state works

---

### Release

- Git tag created
- Release notes written
- Package published

---

# E. “Practice Smell” Diagnostic (10-Second Test)

If a project feels like a practice exercise, it usually fails several of the following checks.

Ask these questions quickly:

### Structure

- Does the project have a clear directory structure?

---

### Installation

- Can it be installed in **one command**?

---

### Documentation

- Does the README explain usage clearly?

---

### Interface

- Are inputs and outputs predictable?

---

### Observability

- Are logs present?

---

### Error Handling

- Are error states handled cleanly?

---

### UI

- Is spacing consistent?

---

### Release

- Is there a version number?

---

### Future

- Is there a roadmap?

---

### Trust

- Is there any mention of security or privacy?

---

### The Rule

If a project fails most of these checks, it will **look like a learning project**.

If it passes most of them, it will **look like a real product**, even if the codebase is small.

This is the power of maturity signals.

They transform perception.

And perception is often the difference between a simple demo and a professional system.
