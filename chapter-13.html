14  Logging and Observability: Making Systems Understandable

When a small script runs on a developer’s machine, understanding what the system is doing is usually straightforward. If something goes wrong, the developer can print a few variables, inspect the code, or rerun the program step by step.

But once software grows into a real system—running on servers, interacting with databases, handling multiple requests simultaneously—the situation changes dramatically.

At that point, the developer can no longer observe the system directly. The code may still execute correctly, but its behavior becomes hidden behind layers of infrastructure and time.

Professional systems solve this problem through logging and observability.

These mechanisms allow developers and operators to understand what the system is doing, even when they cannot see it directly.

Without them, a system becomes opaque. When something fails, no one knows why.

With them, the system becomes explainable.

14.0.1 The Purpose of Logging

Logging is the process of recording important events that occur while the system is running.

Each log entry captures a moment in the life of the system. It may record an incoming request, the start of a database query, the completion of an operation, or an unexpected error.

Over time, these records form a timeline that reveals how the system behaved.

In practice projects, logging is often minimal or absent. Developers may print debugging messages while developing the code and remove them once the program appears to work.

Professional systems treat logging differently. Instead of temporary debugging statements, they maintain a structured record of significant events throughout the lifetime of the application.

These records allow developers to reconstruct what happened during execution—even long after the event has occurred.

14.0.2 Structured Logs

In mature systems, logs are not merely lines of text. They are structured records.

A structured log contains clearly defined fields that describe the event being recorded. Instead of writing free-form messages, the system captures key attributes in a consistent format.

Typical fields might include the time of the event, the type of operation, the component responsible, and relevant identifiers associated with the request.

This structure allows logs to be searched, filtered, and analyzed automatically. Monitoring systems can aggregate thousands of events and identify patterns that would be impossible to detect through manual inspection.

Structured logging transforms logs from simple messages into a dataset describing the system’s behavior.

14.0.3 Trace Identifiers

As systems grow more complex, individual operations often pass through multiple components.

For example, a request may enter through an API gateway, travel through application services, interact with a database, and eventually produce a response. Each of these steps may generate log entries.

Without additional context, it becomes difficult to determine which log entries belong to the same request.

To solve this problem, professional systems introduce trace identifiers.

A trace identifier is a unique value assigned to a request as it enters the system. This identifier travels with the request through every component involved in processing it.

Each log entry related to that request includes the same identifier.

This simple mechanism allows developers to reconstruct the entire path of a request across the system. By filtering logs using the trace identifier, they can observe every stage of the operation and quickly identify where problems occurred.

14.0.4 Recording Key Events

Not every line of code needs to produce a log entry. Excessive logging can create noise that makes important information harder to find.

Instead, professional systems focus on recording key events—moments that reveal the state and progress of the system.

Examples of such events include:

  • the arrival of a request

  • the beginning or completion of a significant operation

  • interactions with external services

  • changes to persistent data

  • unexpected errors or warnings

These events create a meaningful narrative of the system’s activity.

By observing them, developers can understand how the system behaves under different conditions.

14.0.5 Knowing When to Log

An effective logging strategy requires thoughtful decisions about when logging is useful.

Events that represent important transitions in the system’s workflow should usually be recorded. These transitions help developers understand how the system moves from one stage to another.

In contrast, extremely low-level details—such as the execution of every small function—rarely provide useful information in production environments.

A balanced approach focuses on visibility without overwhelming the logging system with unnecessary detail.

The goal is not to record everything but to capture the events that explain the system’s behavior.

14.0.6 Knowing What to Log

Equally important is deciding what information should accompany each log entry.

A useful log entry provides enough context to understand the event without requiring additional investigation.

This context may include identifiers associated with the request, parameters that influenced the operation, or the result produced by a particular component.

At the same time, professional systems must remain cautious about recording sensitive information. Data such as passwords, authentication tokens, or personal details should never appear in logs.

Logging therefore requires a balance between visibility and responsibility.

14.0.7 From Opaque Systems to Observable Systems

The introduction of logging and observability changes the nature of a system.

Without these mechanisms, the system behaves like a sealed box. Inputs enter, outputs emerge, but the internal process remains hidden.

With proper observability, the system becomes transparent. Developers can observe how requests move through the architecture, how components interact, and where failures occur.

This transparency dramatically improves the ability to maintain and evolve the system over time.

In practice projects, such capabilities often appear unnecessary. The developer controls the environment and can inspect the code directly.

In professional systems, however, visibility becomes essential.

A system that cannot explain its own behavior cannot be trusted to operate reliably in the long term.

Logging and observability therefore represent more than engineering tools. They are foundational elements that allow software to remain understandable even as it grows in scale and complexity.
