22  Versioning, Changelogs, and Release Rhythm

As software evolves, it accumulates changes.

Features are added, bugs are fixed, behaviors are improved, and sometimes existing functionality is modified. From the perspective of the developer, these changes are part of the natural process of building and refining the system.

However, for users and collaborators, changes can be confusing unless they are communicated clearly.

Professional projects therefore treat change itself as something that must be managed and explained. They introduce mechanisms that make the evolution of the system visible and understandable.

Three of the most common mechanisms are versioning, changelogs, and release notes.

Together, these tools allow a project to communicate how it is changing over time.

22.0.1 Version Numbers as Signals

A version number is more than a label attached to a release. It is a signal that helps users understand what kind of changes they should expect.

Professional projects frequently follow a structured versioning scheme known as Semantic Versioning, often abbreviated as SemVer.

In this scheme, version numbers follow a three-part format:

MAJOR.MINOR.PATCH

Each part communicates a different level of change.

A patch update typically indicates small fixes that do not alter existing behavior. A minor update introduces new functionality while preserving compatibility with previous usage. A major update signals that significant changes may alter existing behavior or require users to adapt their workflows.

Because the meaning of each number is consistent, users can interpret the impact of an update without reading the entire codebase.

Version numbers therefore act as a language through which a project communicates its evolution.

22.0.2 Changelogs as Historical Records

While version numbers provide signals, they do not describe the details of what changed.

That role belongs to the changelog.

A changelog is a structured record of changes introduced in each release. Instead of forcing users to inspect commit histories or code differences, the changelog summarizes the important updates in a readable format.

Typical entries might include new features, bug fixes, performance improvements, and changes to existing behavior.

By organizing these entries by version, the changelog becomes a historical record of the project’s development.

Users can quickly review how the system has evolved and determine whether a new release affects their usage.

22.0.3 Release Notes as Communication

Release notes extend this idea further by framing changes in a way that is meaningful to users.

While a changelog may list technical updates, release notes often explain why the changes matter. They may highlight the most important improvements, describe new capabilities, or provide guidance on how to adopt updated features.

In this sense, release notes are not merely technical documentation.

They are a form of communication.

They translate internal development activity into information that users can understand and appreciate.

22.0.4 Establishing a Release Rhythm

Another aspect of professional software development is the concept of a release rhythm.

Practice projects often release updates irregularly. Changes accumulate until the developer decides to publish a new version.

Professional projects tend to follow more predictable patterns.

Updates may be released at regular intervals, or the team may follow defined milestones for major improvements. Even when the exact schedule varies, the project communicates its progress consistently.

This rhythm helps users understand when to expect improvements and how actively the project is being maintained.

A steady release rhythm signals that the system is alive and evolving.

22.0.5 Speaking to the Outside World

Many practice projects remain inward-facing. The developer understands the changes because they wrote the code, but the project does not explain itself to others.

Professional systems recognize that software exists within a community of users, collaborators, and stakeholders.

These audiences need clear signals about how the system evolves.

Version numbers provide a quick indication of change. Changelogs offer a structured history of development. Release notes translate technical updates into meaningful communication.

Together, these elements allow the project to speak to the outside world.

This ability to communicate evolution clearly is an important aspect of maturity.

A professional project does not simply change over time.

It explains those changes in a way that others can follow and trust.

22.1 实践指南：语义化版本发布SOP

有效的版本管理需要清晰的规则和可重复的流程。以下是一个实用的版本发布框架。

22.1.1 语义化版本规范（SemVer）

版本号格式：MAJOR.MINOR.PATCH

MAJOR（主版本号）： - 当进行不兼容的API修改时递增 - 示例：1.0.0 → 2.0.0 - 意味着：现有代码可能需要修改

MINOR（次版本号）： - 当添加向后兼容的功能时递增 - 示例：1.0.0 → 1.1.0 - 意味着：现有代码无需修改

PATCH（修订号）： - 当进行向后兼容的问题修复时递增 - 示例：1.0.0 → 1.0.1 - 意味着：纯修复，无功能变化

22.1.2 发布流程SOP

第一步：变更收集

  1. 从上次发布开始，收集所有变更
  2. 分类为：新增功能、Bug修复、破坏性变更
  3. 确定影响的范围和重要性

第二步：版本号决策

  1. 是否有破坏性变更？→ 递增MAJOR
  2. 是否有新功能？→ 递增MINOR
  3. 是否只有修复？→ 递增PATCH
  4. 记录决策理由

第三步：更新文件

  1. 更新package.json或pom.xml中的版本号
  2. 更新CHANGELOG.md
  3. 更新README.md（如果需要）
  4. 更新文档中的版本引用

第四步：创建发布

  1. 创建Git标签：git tag -a v1.2.3 -m "Release v1.2.3"
  2. 推送标签：git push origin v1.2.3
  3. 构建发布包
  4. 发布到包管理器（npm, PyPI等）

第五步：发布通知

  1. 在GitHub创建Release
  2. 撰写Release Notes
  3. 通知用户（邮件、博客、社交媒体）
  4. 监控反馈

22.5 本章总结

版本管理不仅是技术实践，更是与用户沟通的方式。

通过语义化版本、清晰的CHANGELOG和详细的Release Notes，我们让系统的演化变得透明和可理解。

好的版本管理让用户能够信任系统，知道何时升级、如何升级以及升级会带来什么影响。

下一章，我们将探讨路线图。如何向用户展示系统的未来方向？路线图如何帮助管理期望和指导开发？

实践建议：

  1. 自动化发布：尽可能自动化发布流程，减少人为错误。
  2. 保持CHANGELOG更新：每次提交都更新CHANGELOG的Unreleased部分。
  3. 提前预告：对于破坏性变更，提前通知用户，给出迁移时间。
  4. 版本冻结：在发布前几天冻结代码，只允许修复，避免最后一刻变更。

立即行动：

为你的项目创建一个CHANGELOG.md文件，并制定下次发布的计划。确保你的版本号遵循语义化版本规范。
