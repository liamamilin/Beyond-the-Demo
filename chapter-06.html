7  Domain Models: From Dictionaries to Entities and Relationships

In many practice projects, data begins as something simple and flexible. A developer may store information in dictionaries, lists, or loosely structured JSON objects. These formats are convenient because they allow rapid experimentation without requiring strict definitions.

At the early stages of development, this flexibility is valuable. It enables ideas to move quickly from concept to working code.

However, as a project grows, this informal approach to data begins to reveal its limitations. When the system’s data is only loosely defined, the structure of the entire application becomes unstable. Logic spreads across multiple places, assumptions become hidden, and changes become increasingly difficult.

Professional systems address this problem by developing a domain model.

A domain model is not simply a way of storing data. It is a representation of the core concepts that exist within the system and the relationships between them. Once these concepts are clearly defined, the rest of the system can organize itself around them.

In this sense, the domain model becomes the skeleton of the software.

7.0.1 From Flexible Structures to Defined Entities

Practice projects often treat data as generic containers. A developer might store information like this:

A dictionary representing a transaction.
A list representing tasks.
A JSON object representing user data.

These structures work well for quick experimentation, but they do not communicate meaning clearly. A dictionary might contain many fields, yet nothing explicitly states what the object represents within the system.

As a result, the logic that interprets the data becomes scattered across different parts of the codebase.

Domain models solve this problem by introducing entities—clearly defined objects that represent meaningful concepts in the system.

An entity is something that has a distinct identity within the system. It can change over time while still representing the same conceptual object.

Examples might include:

  • a user account

  • a transaction

  • a task

  • a project

Each entity represents a stable concept around which behavior and rules can be organized.

Once entities are introduced, the system begins to describe itself in terms of real concepts rather than generic containers.

7.0.2 Value Objects: Describing Attributes with Meaning

Not all data in a system needs its own identity.

Some pieces of information exist only to describe attributes of an entity. These pieces are often referred to as value objects.

A value object does not represent a distinct object in the system. Instead, it represents a meaningful value associated with an entity.

For example, an amount of money, a timestamp, or a geographic location may function as value objects. Their significance lies not in their identity but in the value they hold.

By distinguishing between entities and value objects, the system gains clarity. Developers can immediately understand which parts of the data represent core objects and which parts simply describe those objects.

This separation helps prevent unnecessary complexity while preserving meaningful structure.

7.0.3 Aggregates: Grouping Related Concepts

As systems grow, entities rarely exist in isolation. They often form clusters of related objects that must be managed together.

An aggregate represents such a cluster. It defines a boundary within which certain entities and value objects interact closely.

Within this boundary, one entity usually acts as the primary reference point for the group. Changes to related objects pass through this central entity, ensuring that the system maintains consistent rules.

The purpose of aggregates is not to introduce abstract theory but to maintain coherence. By grouping related concepts together, the system prevents scattered logic and uncontrolled interactions between components.

In simpler terms, aggregates help ensure that related pieces of data move together in predictable ways.

7.0.4 How Data Models Shape the System

Once the domain model becomes clear, an interesting transformation occurs.

Many architectural decisions begin to follow naturally from the structure of the data.

If entities and relationships are well defined, storage strategies become easier to design. Validation rules become clearer. Interfaces begin to reflect the same conceptual structure. Even user interfaces often mirror the organization of the underlying model.

For example, if a system revolves around transactions and accounts, many features will naturally center around these entities. Interfaces may allow users to create transactions, review account histories, or analyze spending patterns.

In this way, the domain model influences not only how data is stored but also how the entire system behaves.

When the data model is weak or ambiguous, the software tends to feel fragmented. When the model is strong, the system gains coherence.

7.0.5 The Skeleton of the Software

从非正式数据结构到清晰领域模型的转变，标志着项目成熟化的重要一步。

系统不再将数据视为灵活容器的集合，而是开始认识自己的内部概念。实体定义了系统的主要对象。值对象提供了有意义的描述。聚合将相关元素组织成连贯的结构。

这些元素共同构成了软件的骨架。

一旦这个骨架存在，系统的其余部分就可以围绕它以更大的稳定性生长。新功能自然集成，职责更容易分配，系统开始反映它所代表的领域。

在这个阶段，项目不再感觉像一个快速的实验。它开始像一个具有内部结构的系统，能够支持长期的演进。

没有良好领域模型的系统就像一个没有骨架的生物——它可能暂时存活，但无法承受重量和压力。随着复杂性增加，缺乏结构的数据模型会变得越来越脆弱，难以维护和扩展。

而一个清晰的领域模型为系统提供了坚实的结构基础。就像建筑物的框架一样，它定义了系统的整体形状，并为其他组件提供了附着点。

7.1 Practical Guide: Domain Modeling Steps

领域建模不是艺术，而是一门可以学习和实践的技艺。以下是一个实用的建模框架。

7.1.1 领域建模SOP

第一步：识别核心概念

  1. 列出系统处理的所有事物
  2. 识别哪些是独立的对象（实体）
  3. 识别哪些是描述性属性（值对象）

第二步：定义关系

  1. 识别概念之间的关系（一对一、一对多、多对多）
  2. 为每个关系命名（例如：“拥有”、“包含”、“属于”）
  3. 确定关系的方向和基数

第三步：定义聚合

  1. 识别哪些概念总是作为一组被操作
  2. 为每个聚合定义根实体
  3. 确保只有聚合根可以访问聚合内部的其他对象

第四步：验证

  1. 模型是否反映了真实的业务领域？
  2. 是否有遗漏的概念或关系？
  3. 模型是否足够简单以避免过度设计？

7.1.2 实体 vs 值对象：决策树

这个对象需要身份吗？
├─ 是 → 实体（Entity）
│   └─ 它有生命周期吗？
│       ├─ 是 → 实体
│       └─ 否 → 重新考虑是否需要身份
└─ 否 → 值对象（Value Object）
    └─ 它可以被多个实体共享吗？
        ├─ 是 → 值对象
        └─ 否 → 可能是实体

7.2 Chapter Summary

领域模型是软件的骨架。它将模糊的数据结构转化为明确定义的概念和关系，为整个系统提供结构基础。

通过实体、值对象和聚合，系统开始反映真实的业务领域。这种反映不仅使代码更易理解，也使系统更稳定、更易维护。

一个好的领域模型不是凭空出现的，它来自于对业务领域的深入理解和持续的重构。它是一个演化的过程，而不是一次性的设计。

下一章，我们将探讨如何组织代码的执行流程。当控制流从条件分支演变为清晰的管道时，系统的可读性和可维护性会如何提升？

实践建议：

  1. 从需求开始：不要直接从数据库表开始建模。先理解业务领域和需求。
  2. 迭代改进：领域模型不需要一次性完美。随着理解的深入，持续改进它。
  3. 使用通用语言：确保技术团队和业务团队使用相同的术语来描述领域概念。
  4. 文档化模型：保持领域模型的文档更新，包括实体图和关系说明。

立即行动：

选择你的项目中的一个核心功能，尝试画出涉及的所有实体和它们的关系。然后，问自己：这个模型是否清晰反映了业务领域？
