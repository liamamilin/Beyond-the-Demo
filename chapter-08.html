9  Interfaces: The Three Faces of a Professional System

As software matures, one of the most visible changes occurs at its boundaries—the points where users or other systems interact with it.

Practice projects usually expose only a single entry point. A developer might run a script directly, call a function manually, or interact with a minimal interface created only for demonstration.

This approach works during experimentation, because the primary user of the system is the developer who built it.

Professional systems, however, operate in a different environment. They are expected to serve multiple types of users and integrate with other systems. As a result, they often present multiple interfaces through which the same underlying functionality can be accessed.

These interfaces typically take three common forms:

  • a command-line interface (CLI)

  • an application programming interface (API)

  • a graphical user interface (UI)

Each interface represents a different way of interacting with the same system.

Together, they form the three faces of a professional software system.

9.0.1 The Single Entry Point of Practice Projects

In early-stage projects, interaction with the system is usually direct and informal.

A developer might run a file from the terminal, execute a function inside a development environment, or pass input directly through a simple script. The interface is minimal because the system is not designed for repeated use by others.

This single entry point reflects the purpose of a practice project: to verify that the logic works.

However, this approach creates limitations. The system is difficult to reuse in other contexts. Automation becomes awkward. External tools cannot interact with the software easily.

In essence, the functionality exists, but it remains locked inside the code.

Professional systems remove this limitation by separating the core logic from the interfaces that expose it.

9.0.2 The Command-Line Interface

The command-line interface is often the simplest way to expose a system in a structured and reusable manner.

A CLI allows users to interact with the system through commands and arguments. Instead of executing code directly, users invoke clearly defined operations.

For example, a task management system might expose commands such as creating tasks, listing existing tasks, or generating reports.

The key characteristic of a well-designed CLI is predictability. Commands follow consistent patterns. Arguments are structured logically. Error messages explain what went wrong and how to correct it.

Because of this predictability, CLIs are widely used not only by individuals but also by scripts and automation tools.

In many professional systems, the CLI becomes a powerful control surface that allows the software to be integrated into larger workflows.

9.0.3 The Application Programming Interface

While CLIs are designed primarily for human interaction, APIs allow other software systems to communicate directly with the application.

An API defines structured endpoints through which operations can be performed programmatically. Instead of typing commands, another system sends requests that follow a predefined format.

These requests may retrieve information, create new records, update existing data, or trigger processing tasks.

The presence of an API transforms a program from a standalone application into a service. Other tools, platforms, or applications can incorporate its functionality as part of a larger ecosystem.

To achieve this integration, the API must maintain strict consistency. Inputs and outputs must follow stable formats, and errors must be communicated clearly.

Predictability is even more critical here, because other systems rely on the API’s behavior remaining stable over time.

9.0.4 The Graphical User Interface

The graphical user interface provides the most direct and accessible form of interaction for many users.

A UI presents the system visually, allowing users to navigate through screens, forms, and interactive components. It translates the system’s functionality into a layout that can be understood quickly without reading technical documentation.

A professional interface goes beyond simply displaying information. It organizes actions logically, provides feedback during operations, and communicates system state clearly.

Loading indicators, confirmation messages, and error notifications all contribute to a sense that the system is responsive and reliable.

Although the UI may be the most visible part of the software, it usually sits on top of the same core logic used by the CLI and API.

This separation ensures that improvements to the interface do not require rewriting the underlying system.

9.0.5 Consistency Across Interfaces

The presence of multiple interfaces does not mean that each one behaves differently.

On the contrary, professional systems strive to maintain consistent behavior across all interfaces.

A command executed through the CLI should produce the same effect as a request made through the API or an action performed in the graphical interface. The differences lie only in how the request is expressed, not in what the system ultimately does.

This consistency reinforces the perception that all interfaces connect to a single coherent system.

It also simplifies development. When the core logic is independent of the interface layer, each interface becomes a thin translation layer that converts user actions into system operations.

9.0.6 Interfaces as the System’s Public Surface

Interfaces represent the public surface of a system.

They are the points where the internal architecture becomes visible to the outside world. If the interfaces are inconsistent, confusing, or unpredictable, the entire system appears unreliable.

Conversely, when interfaces are well designed, users and developers immediately sense that the system has been carefully constructed.

A project that exposes only a single ad hoc entry point feels experimental. A project that presents multiple consistent interfaces feels like a platform.

This transformation—from a single script to a system with several coherent entry points—marks another step in the journey from practice project to professional software.

9.1 Practical Guide: Multi-Interface System Design SOP

Multi-interface system design is the key to transforming a single entry point into a professional platform. The following is a practical interface design framework to help you build a consistent and powerful interface system.

9.1.1 Interface Type Classification

CLI (Command Line Interface): - Target users: Developers, operations personnel, automation scripts - Features: Fast, programmable, easy to integrate - Use cases: Batch operations, automation tasks, development debugging

API (Application Programming Interface): - Target users: Other systems, third-party applications - Features: Standardized, programmable, cross-platform - Use cases: System integration, microservices architecture, third-party integration

UI (Graphical User Interface): - Target users: End users, non-technical personnel - Features: Intuitive, visual, easy to understand - Use cases: Daily use, data display, complex operations

# Core logic independent of interfaces
class TaskService:
    """Core logic for task management"""

    def create_task(self, title: str, description: str = "", priority: str = "medium") -> Task:
        """Core logic for creating tasks"""
        task = Task(
            id=generate_id(),

            status="pending",
            created_at=datetime.utcnow()
        )
        self.repository.save(task)
        return task

    def get_task(self, task_id: str) -> Task:
        """Core logic for retrieving tasks"""
        return self.repository.find(task_id)

# CLI interface layer
class TaskCLI:
    """Command line interface"""

    def __init__(self, service: TaskService):
        self.service = service

    def create(self, title: str, description: str = "", priority: str = "medium"):
        """CLI create task command"""
        try:
            task = self.service.create_task(title, description, priority)
            print(f"✓ 任务创建成功: {task.id}")
            print(f"  标题: {task.title}")
            print(f"  优先级: {task.priority}")
        except ValueError as e:
            print(f"✗ 错误: {e}")
            sys.exit(1)

# API interface layer
class TaskAPI:
    """RESTful API接口"""

    def __init__(self, service: TaskService):
        self.service = service

    def create_task(self, request):
        """API创建任务端点"""
        data = request.json

        # 验证输入
        if not data.get("title"):
            return {
                "code": "VALIDATION_ERROR",
                "message": "标题不能为空"
            }, 400

        # 调用核心逻辑
        task = self.service.create_task(
            title=data["title"],
            description=data.get("description", ""),
            priority=data.get("priority", "medium")
        )

        return {
            "id": task.id,
            "title": task.title,
            "description": task.description,
            "priority": task.priority,
            "status": task.status,
            "created_at": task.created_at.isoformat()
        }, 201

Unified authentication mechanism：

# 认证中间件
def authenticate(func):
    def wrapper(*args, **kwargs):
        token = get_auth_token()
        user = verify_token(token)
        if not user:
            raise UnauthorizedError()
        return func(*args, **kwargs)
    return wrapper

9.1.3 Interface consistency checklist

Naming consistency： - [ ] CLI命令使用动词（create, list, delete） - [ ] API端点使用名词（/tasks, /users） - [ ] UI按钮使用动作描述（创建任务、删除任务） - [ ] 所有接口使用相同的术语

Behavior consistency： - [ ] 所有接口执行相同的操作逻辑 - [ ] 相同的输入产生相同的输出 - [ ] 错误处理机制统一 - [ ] 认证方式一致

Response format consistency： - [ ] API使用统一的JSON格式 - [ ] 错误码标准化 - [ ] 成功响应结构一致 - [ ] 时间戳使用ISO 8601格式

Authentication and authorization： - [ ] 所有接口使用相同的认证机制 - [ ] 权限检查在接口层统一处理 - [ ] 敏感操作需要额外授权 - [ ] API密钥管理安全

9.2 Chapter Summary

多接口系统设计是将单一入口转化为专业平台的关键。CLI、API和UI三种接口为不同类型的用户提供了访问系统的不同方式。

良好的接口设计遵循核心逻辑与接口分离的原则，确保所有接口的一致性。Unified response format、错误码和认证机制提升了系统的专业性。

一个拥有多个一致接口的系统，不再是孤立的程序，而是一个可扩展的平台。

下一章，我们将探讨接口契约。如何明确定义输入输出，让接口行为可预测？

实践建议：

  1. 核心逻辑优先：先实现核心业务逻辑，然后再添加接口层。这样更容易维护和测试。
  2. 接口契约先行：在设计接口之前，先定义清晰的契约（OpenAPI规范）。
  3. 统一标准：所有接口遵循相同的命名、响应格式和错误处理模式。
  4. 文档驱动：接口文档应该与代码同步更新，保持准确性。
