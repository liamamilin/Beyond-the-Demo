6  Naming: From Temporary Variables to a Domain Language

At the deepest level, software is simple. It consists of three fundamental elements:

Objects + Actions + States

This triad represents the essence of what software does: it manipulates objects through actions, changing their states over time. Every function, class, and variable in a codebase maps to one of these three concepts.

When developers understand this philosophy, naming becomes straightforward:

  • Objects are named with nouns: Transaction, Account, User, Order
  • Actions are named with verbs: create, update, validate, process
  • States are named with adjectives or state nouns: active, pending, complete, isDeleted, hasPermission

This simple framework eliminates most naming uncertainty. Instead of asking “what should I call this?”, developers ask “what does this represent: an object, an action, or a state?”

The difference between a practice project and a professional system often begins with this question. Practice projects name things arbitrarily. Professional systems name things intentionally, using the object-action-state philosophy as their foundation.

This chapter explores how to apply this philosophy systematically to build a complete, coherent naming vocabulary.

6.0.1 The Object-Action-State Philosophy

Every piece of software can be understood through this triad. A system processes objects by performing actions that transition their states.

Consider an e-commerce application:

  • Objects: ShoppingCart, Product, PaymentMethod, CustomerProfile
  • Actions: addToCart(), processPayment(), calculateTotal(), validateInventory()
  • States: cartEmpty, paymentPending, inventoryAvailable, userAuthenticated

These names are not arbitrary. They directly express what each component represents in the system’s domain model.

When naming follows this framework, code becomes self-documenting. A developer reading cart.processPayment() immediately understands that a cart object (noun) is undergoing a payment process (verb) which will transition the cart’s state from something to something else.

This clarity comes not from extensive comments or documentation, but from the systematic application of a simple naming philosophy.

6.0.2 Object Naming System

Objects are the nouns of software. They represent the things that exist within the system’s domain: data entities, concepts, and structures that have identity and behavior.

Professional systems distinguish between several types of objects, each with its own naming convention.

Entities

Entities are objects that have identity and lifecycle. They represent the core concepts in the domain:

  • Customer, Order, Product, Invoice, Account
  • Session, Transaction, Subscription, Membership

Entity names should be singular nouns that clearly indicate what the object represents. They capture the essence of the domain concept.

Value Objects

Value objects represent values or measurements that don’t have independent identity. They are defined by their attributes:

  • EmailAddress, Money, Address, PhoneNumber
  • Color, Coordinate, Duration, Percentage

Value object names describe what they represent, emphasizing their value nature rather than their identity.

Services

Services represent operations or capabilities that don’t naturally belong to a specific entity:

  • PaymentProcessor, EmailService, AuthenticationProvider, NotificationService
  • DataValidator, FileExporter, CacheManager, RateLimiter

Service names typically end with capability descriptors like Processor, Service, Provider, or Manager, clearly indicating that the object provides functionality rather than representing data.

The distinction between these object types is important because it prevents confusion in the codebase. When developers see EmailAddress, they immediately understand it’s a value object. When they see EmailService, they know it provides email functionality.

This clarity allows the system’s domain language to emerge naturally from the naming choices.

6.0.3 Action Naming System

Actions are the verbs of software. They represent what the system does: operations, transformations, and behaviors that manipulate objects or transition states.

Professional systems classify actions into four distinct categories, each with specific naming patterns.

Command Actions

Command actions perform state changes or side effects. They modify the system:

  • createUser(), updateProfile(), deleteAccount(), archiveRecord()
  • sendEmail(), processPayment(), publishArticle(), expireSession()

Command names use strong verbs that clearly indicate the action being performed. They typically follow the pattern verbObject() where the verb describes what happens and the object describes what is affected.

Query Actions

Query actions retrieve information without changing state. They ask questions of the system:

  • getUserById(), findOrdersByDate(), getAvailableInventory(), calculateTotal()
  • searchProducts(), fetchProfile(), listSubscriptions(), validateCredentials()

Query names use verbs that emphasize retrieval or computation: get, find, fetch, list, search, calculate, validate. These names make it clear that the function returns information rather than modifying state.

Boolean Actions

Boolean actions return true/false answers. They test conditions:

  • isValid(), hasPermission(), canEdit(), isAuthenticated()
  • exists(), isEmpty(), isComplete(), hasAccess()

Boolean action names typically start with is, has, can, or similar interrogative verbs. This naming convention immediately signals that the return value is a boolean, making the code more readable and predictable.

Conversion Actions

Conversion actions transform one type of object into another:

  • toDTO(), toEntity(), toJson(), toString()
  • convertToCurrency(), mapToView(), serialize(), deserialize()

Conversion names use verbs that indicate transformation: to, convert, map, serialize, transform. They explicitly show the source and destination types, making the conversion process transparent.

By categorizing actions this way, developers can immediately understand the nature of any function or method just from its name. This eliminates ambiguity and makes the codebase more predictable and maintainable.

6.0.4 State Naming System

States are the adjectives and state nouns of software. They represent conditions, phases, and attributes that describe objects or the system as a whole.

Professional systems use four main patterns for naming states, depending on the type of condition being represented.

Boolean States

Boolean states represent simple true/false conditions:

  • isActive, isDeleted, isVisible, isPublished
  • hasPermission, hasSubscription, hasAccess, isVerified

Boolean state names typically use the is or has prefix, making the boolean nature immediately obvious. These names are self-documenting: if (user.isActive) reads like natural language.

Enumeration States

Enumeration states represent distinct phases or categories:

  • OrderStatus: pending, processing, shipped, delivered, cancelled
  • UserStatus: active, inactive, suspended, deleted
  • PaymentStatus: authorized, captured, refunded, failed

Enumeration names use clear, descriptive terms that represent the possible states. They avoid technical jargon in favor of domain-relevant language: shipped rather than state_3.

Numerical States

Numerical states represent measurable quantities or counters:

  • retryCount, remainingAttempts, totalAmount, balance
  • priorityLevel, versionNumber, timestamp, duration

Numerical state names describe what the number represents rather than just calling it count or value. The name provides context: retryCount clearly indicates this counts retries, not something else.

Composite States

Composite states combine multiple pieces of information:

  • shippingAddress, billingProfile, contactInformation
  • paymentDetails, subscriptionPlan, userPreferences

Composite state names use noun phrases that group related attributes together. They create natural containers for related data, making the code more organized and the relationships clearer.

When state naming follows these patterns, reading code becomes intuitive. if (order.status === OrderStatus.shipped) immediately tells you what you’re checking, without needing to look up documentation or decode cryptic state names.

6.1 Practical Guide: Naming Standards and Review

Good naming follows from a systematic philosophy. The object-action-state framework provides a foundation for making consistent naming decisions.

6.1.1 Naming Framework Application

Step 1: Identify the Category

For every identifier you create, ask: Is this an object, an action, or a state?

# Example: Analyzing code elements

user                    # Object (Entity)
processPayment()        # Action (Command)
isActive               # State (Boolean)
UserStatus            # Object (Enumeration)
calculateTotal()       # Action (Query)
retryCount            # State (Numerical)

Step 2: Apply the Appropriate Pattern

For Objects: - Use singular nouns - Distinguish between entities, value objects, and services - Choose domain-relevant terms

For Actions: - Use strong verbs - Classify as command, query, boolean, or conversion - Follow the naming convention for the type

For States: - Use adjectives or descriptive nouns - Indicate the state type clearly (boolean, enumeration, numerical, composite) - Make the condition explicit

6.1.2 Naming Consistency Checklist

Object Naming: - [ ] All entities use singular, domain-relevant nouns - [ ] Value objects describe their value nature - [ ] Services end with capability descriptors (Service, Processor, Manager) - [ ] No generic names (data, info, item, thing)

Action Naming: - [ ] Commands use strong, descriptive verbs (create, update, delete) - [ ] Queries use retrieval verbs (get, find, fetch, list, search) - [ ] Boolean actions start with is/has/can - [ ] Conversions indicate transformation (to, convert, map)

State Naming: - [ ] Boolean states use is/has prefix - [ ] Enumeration states use domain-relevant terms - [ ] Numerical states describe what the number represents - [ ] Composite states group related attributes

6.1.3 Naming Review Process

1. The Three-Second Test

Can you understand what a name represents within three seconds? If not, the name needs improvement.

2. The Object-Action-State Mapping

For each identifier, verify it clearly fits into one of the three categories:

  • If it’s an object: Is it a noun? Does it represent a domain concept?
  • If it’s an action: Is it a verb? Does it indicate what it does?
  • If it’s a state: Is it an adjective or state noun? Does it describe a condition?

3. The Consistency Check

Are similar concepts named consistently across the codebase? Do you use the same terminology for the same idea everywhere?

6.1.5 Naming Template Library

The following templates provide detailed patterns for naming different types of code elements based on the object-action-state philosophy.

6.1.5.1 Object Naming Templates

Descriptive Objects (Domain Concepts) - Pattern: {DomainConcept}{OptionalModifier} - Examples: ActiveUser, PendingOrder, ArchivedDocument, PremiumCustomer, VerifiedEmail

Technical Objects (Infrastructure) - Pattern: {DomainConcept}{TechnicalSuffix} - Examples: UserRepository, PaymentGateway, DataValidator, FileExporter, CacheManager

Collection Objects - Pattern: {ElementName}{CollectionType} - Examples: UserList, OrderCollection, ProductSet, ItemMap, EventQueue - Alternative: {ElementName}s (e.g., Users, Orders, Products)

Abstract Interfaces and Base Classes - Pattern: I{InterfaceName} or {Concept}Base or {Concept}Abstract - Examples: IUserService, UserRepositoryBase, PaymentProcessorAbstract

Factory and Builder Objects - Pattern: {TargetType}{Factory|Builder} - Examples: UserFactory, OrderBuilder, PaymentProcessorBuilder, SessionFactory

Configuration Objects - Pattern: {TargetType}Config or {TargetType}Settings - Examples: DatabaseConfig, PaymentSettings, EmailConfig, ApplicationSettings

6.1.5.2 Action Naming Templates

Command Actions (State Changes) - Create: create{Target}(), add{Element}(), register{Entity}() - Update: update{Target}(), modify{Target}(), change{Attribute}() - Delete: delete{Target}(), remove{Element}(), archive{Target}() - Process: process{Target}(), execute{Operation}(), perform{Action}() - Examples: createUser(), updateProfile(), deleteAccount(), archiveRecord(), processPayment()

Query Actions (Information Retrieval) - Simple Get: get{Target}(), get{Target}By{Criteria}() - Find: find{Target}(), find{Target}By{Criteria}() - Search: search{Target}(), search{Target}By{Criteria}() - List: list{Target}(), list{Target}By{Criteria}() - Fetch: fetch{Target}(), fetch{Target}From{Source}() - Calculate: calculate{Metric}(), compute{Result}() - Validate: validate{Input}(), verify{Condition}() - Examples: getUserById(), findOrdersByDate(), searchProductsByName(), listActiveUsers(), calculateTotal()

Boolean Actions (Condition Testing) - Is Prefix: is{Condition}(), is{Adjective}() - Has Prefix: has{Property}(), has{Attribute}() - Can Prefix: can{Action}(), should{Action}() - Must Prefix: must{Condition}(), requires{Condition}() - Exists: exists{Target}(), contains{Element}() - Examples: isValid(), hasPermission(), canEdit(), isAuthenticated(), exists()

Conversion Actions (Type Transformation) - To Prefix: to{TargetType}(), to{TargetType}Format() - Convert: convertTo{TargetType}(), convert{Source}To{Target}() - Map: mapTo{TargetType}(), map{Source}To{Target}() - Serialize: serialize(), deserialize(), to{Format}() - Parse: parse{Source}(), parse{Source}To{Target}() - Examples: toDTO(), toJson(), convertToCurrency(), mapToView(), serialize()

6.1.5.3 State Naming Templates

Boolean States (True/False Conditions) - Is Prefix: is{StateAdjective}, is{StateNoun} - Has Prefix: has{PropertyNoun}, has{AttributeNoun} - Can Prefix: can{ActionVerb}, could{ActionVerb} - Should Prefix: should{ActionVerb}, would{ActionVerb} - Must Prefix: must{Condition}, requires{Condition} - Examples: isActive, isDeleted, hasPermission, canEdit, isRequired

Current State (Present Condition) - Pattern: current{StateNoun} - Examples: currentStatus, currentVersion, currentUser, currentLocation, currentSession

Previous/Next State - Pattern: {Previous|Next|Last}{StateNoun} - Examples: previousStatus, nextVersion, lastLogin, nextScheduledTime

State Transitions - Pattern: {Source}To{Target}State or {TransitionType}State - Examples: PendingToActiveState, ArchivedState, TransitioningState, FinalState

Numerical States (Counters and Measurements) - Count: {Property}Count, {Property}Total - Index: {Property}Index, {Property}Position - Amount: {Property}Amount, {Property}Total - Level: {Property}Level, {Property}Degree - Ratio: {Property}Ratio, {Property}Percentage - Examples: retryCount, totalAmount, priorityLevel, completionRatio

Time-Based States - Timestamp: {Event}Timestamp, {Event}Time, {Event}Date - Duration: {Process}Duration, {Activity}Time - Period: {Event}Period, {TimeSpan}Name - Examples: lastLoginTimestamp, processingDuration, sessionTimeout

Enumeration State Values - Pattern: {Status}{StateValue} or simply {StateValue} - Examples: OrderStatus.PENDING, OrderStatus.PROCESSING, UserStatus.ACTIVE, PaymentStatus.AUTHORIZED

Composite States (Grouped Attributes) - Pattern: {Category}{Information|Details|Profile|Config} - Examples: shippingAddress, billingProfile, contactInformation, paymentDetails, userPreferences

6.1.5.4 Special Naming Patterns

Event Objects - Pattern: {Event}{Event|Occurred|Happened} - Examples: UserCreatedEvent, PaymentFailedEvent, OrderShippedEvent

Exception Objects - Pattern: {Condition}{Exception|Error} - Examples: InvalidInputException, PaymentFailedException, AuthenticationError

Test Objects - Pattern: {Target}{Test|TestSuite|TestFixture} - Examples: UserServiceTest, PaymentProcessorTestSuite, UserRepositoryTestFixture

Helper/Utility Objects - Pattern: {Domain}{Helper|Utility|Utils} - Examples: StringHelper, DateUtility, MathUtils

Constants - Pattern: {CONCEPT}_CONSTANT (uppercase with underscores) - Examples: MAX_RETRY_COUNT, DEFAULT_TIMEOUT, API_VERSION

6.1.5.5 Naming Decision Tree

When naming, follow this decision process:

  1. Category Classification: Is it an object, action, or state?
  2. Type Determination: What specific type within the category?
  3. Pattern Selection: Apply the appropriate naming pattern
  4. Consistency Check: Does it match existing naming in the codebase?
  5. Clarity Verification: Is the meaning immediately obvious?

This template library provides a comprehensive set of patterns that can be applied consistently across your entire codebase, creating a coherent and professional naming system.

6.2 Chapter Summary

The object-action-state philosophy provides a systematic foundation for naming in software. Every element in a codebase can be understood as representing an object (noun), an action (verb), or a state (adjective/noun).

By applying this framework consistently, developers create a coherent domain language that makes code self-documenting and intuitive. Objects are named according to their type (entity, value object, service). Actions are classified by their nature (command, query, boolean, conversion). States are distinguished by their category (boolean, enumeration, numerical, composite).

This systematic approach eliminates arbitrary naming decisions and creates a vocabulary that reveals the system’s structure and intent. The result is code that reads like natural language, where developers can understand relationships and responsibilities directly from the names.

The transformation from ad-hoc naming to systematic naming is one of the quiet but powerful shifts that distinguish practice projects from professional systems.

Practice Suggestions:

  1. Start with the Classification: Before naming anything, first determine if it’s an object, action, or state.
  2. Build a Domain Vocabulary: Document the core nouns, verbs, and adjectives that define your system’s domain.
  3. Review for Consistency: Regularly audit your codebase to ensure the same terminology is used consistently.

Immediate Action:

Choose one module in your project and systematically review all identifiers. Classify each as object, action, or state, and verify that the naming follows the appropriate pattern. Rename any that don’t fit the framework.
